# todo-react

Небольшое React-приложение для управления списком задач. Проект сделан как учебная работа и технический эксперимент: цель была не в реализации сложной предметной области, а в отработке архитектуры фронтенд-приложения, локального state management, роутинга и разделения ответственности между слоями.

## Цели проекта

Проект использовался для отработки нескольких инженерных задач:

- построение React-приложения без внешнего роутера и state manager;
- разделение UI, бизнес-логики, страниц, фич и инфраструктурных утилит;
- работа с асинхронным источником данных через единый API-адаптер;
- поддержка двух режимов хранения данных: `json-server` для разработки и `localStorage` для статического деплоя;
- управление производительностью через `useMemo`, `useCallback`, `memo` и локализацию состояния;
- обработка UX-сценариев: поиск, подсветка совпадений, фокус на поле ввода, прокрутка к первой незавершенной задаче, анимация добавления и удаления.

## Архитектура и структура

Проект организован по близкому к Feature-Sliced Design принципу: верхнеуровневый слой приложения отвечает за инициализацию, стили и роутинг; страницы собирают конкретные сценарии; фичи реализуют пользовательские действия; сущность `todo` содержит модель и UI, связанные с задачами; `shared` хранит переиспользуемые UI-компоненты, API-адаптеры, хуки и утилиты.

Такое разделение выбрано для небольшого приложения с перспективой роста. Логика задач не размазана по компонентам страницы: состояние и операции над задачами сосредоточены в модели сущности, а формы добавления, поиска и статистика остаются отдельными фичами, которые используют публичный контекст `todo`.

Инфраструктурные детали вынесены из UI: роутинг, построение URL с учетом `BASE_URL`, API-адаптеры и утилита подсветки текста живут отдельно от компонентов. Это уменьшает связность и позволяет менять источник данных или способ деплоя без переписывания пользовательских сценариев.

## Ключевые решения

### Собственный минимальный роутер

В проекте нет `react-router`. Вместо него реализован небольшой роутер поверх History API: он сопоставляет текущий путь с конфигурацией маршрутов, поддерживает параметры вида `/tasks/:id` и реагирует на `popstate`.

Такой подход оправдан размером проекта: он показывает механику клиентского роутинга без зависимости от полноценной библиотеки. Отдельно обработан `BASE_URL`, чтобы приложение корректно работало при деплое в подпапку GitHub Pages.

### Единый API-контракт для разных источников данных

Доступ к задачам идет через `tasksApi`, который выбирает реализацию по переменной `VITE_STATIC_BACKEND`. Есть два адаптера с одинаковым интерфейсом: серверный через `fetch` к `json-server` и локальный через `localStorage`.

Это отделяет доменную логику от конкретного способа хранения данных. Для разработки можно использовать backend-like окружение с HTTP-запросами, а для статического деплоя оставить приложение полностью автономным.

### Reducer для доменного состояния задач

Список задач управляется через `useReducer`, а не набор независимых `useState`. Операции `SET_ALL`, `ADD`, `TOGGLE_COMPLETE`, `DELETE` и `DELETE_ALL` представлены как явные события изменения состояния.

Для текущего масштаба это не обязательно, но решение делает переходы состояния предсказуемыми и упрощает дальнейшее расширение: например, добавление optimistic updates, undo или обработки ошибок не потребует пересобирать модель с нуля.

### Контекст как граница доменной модели

`TasksProvider` собирает состояние задач, вычисленные данные, refs и обработчики в один контекст. Фичи и компоненты списка получают только готовый публичный API модели.

Это снижает количество prop drilling и фиксирует границу между сборкой сценария и реализацией операций. Компоненты форм не знают, где хранятся задачи и как именно выполняется добавление или удаление.

### Безопасная подсветка поискового запроса

Подсветка совпадений реализована через HTML-разметку, но перед этим текст задачи экранируется, а поисковая строка проходит escaping для регулярного выражения.

Такой подход позволяет использовать `<mark>` для подсветки и при этом не открывать очевидную XSS-дыру через `dangerouslySetInnerHTML`. Это важная деталь, потому что данные задач вводятся пользователем.

### UX-состояния анимаций вынесены в модель

Для добавления и удаления задач используются отдельные состояния `appearingTaskId` и `disappearingTaskId`. Компонент списка не удаляет элемент мгновенно: сначала запускается визуальное состояние, затем reducer обновляет список.

Это компромисс между простотой и качеством интерфейса. Анимация остается декларативной на уровне CSS-классов, а бизнес-операция удаления не смешивается с таймингами внутри отдельного элемента списка.

## Интересные моменты реализации

- Поиск реализован как derived state через `useMemo`, поэтому отфильтрованный список не хранится отдельно и не может рассинхронизироваться с исходными задачами.
- Прокрутка к первой незавершенной задаче построена через вычисление первого подходящего `id` и передачу ref только нужному элементу.
- После добавления задачи сбрасывается поисковый запрос и возвращается фокус в поле ввода, что делает повторный ввод быстрее.
- `RouterLink` сам строит корректный `href` с учетом base path и обновляет историю без перезагрузки страницы.
- Компоненты списка, элемента задачи и статистики мемоизированы. В таком размере приложения выигрыш небольшой, но это демонстрирует контроль над лишними перерендерами.
- SCSS Modules используются для локализации стилей компонентов, а глобальные CSS-файлы оставлены для normalize, переменных, шрифтов и утилитарных классов.

## Что можно улучшить

- Добавить обработку ошибок в модель задач. Сейчас часть API-ошибок не отображается в основном сценарии списка.
- Уточнить контракт `getById`: локальный адаптер возвращает `null`, а серверный адаптер напрямую парсит ответ. Лучше привести поведение адаптеров к одному формату ошибок.
- Добавить тесты для reducer, URL-утилит, подсветки поиска и API-адаптеров.
- Убрать неиспользуемый хук для `localStorage`, если он больше не нужен после появления слоя `shared/api/tasks`.
- Перенести тексты интерфейса и ошибок в константы, если приложение будет расширяться или получит локализацию.
- Для более крупного проекта стоило бы рассмотреть готовый роутер и data-fetching слой, но для учебной цели текущая реализация полезна тем, что показывает внутреннюю механику.

## Демонстрация

Деплой подготовлен через `gh-pages`. Production-сборка использует статический backend на `localStorage`.

GitHub Pages target: https://sergey-bondarenko-dev.github.io/todo-react


## Запуск

Установка зависимостей:

```bash
npm install
```

Запуск dev-сервера:

```bash
npm run dev
```

По умолчанию dev-режим использует серверный API на `http://localhost:3001/tasks`. Для него нужен `json-server` и локальный файл `db.json5`:

```bash
npm run server
```

Для запуска без локального сервера можно включить статический режим через переменную окружения:

```powershell
$env:VITE_STATIC_BACKEND='true'; npm run dev
```

Сборка production-версии:

```bash
npm run build
```

Локальный просмотр production-сборки:

```bash
npm run preview
```
